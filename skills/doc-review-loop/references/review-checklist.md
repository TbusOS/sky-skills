# Reviewer 评审 Checklist · 完整版

每条 checklist 项目都对应一类常见弱点，reviewer 用这个清单逐条过文档。

## 论证质量层

### 1. file:line 证据完整性
- 每个"代码做了 X" / "实测是 Y" / "上游怎么设计"的论断，是否带 file:line 或绝对路径？
- 引用是否可追溯（grep / git show 等命令）？
- 间接引用（"听说 / 别人说"）是否被识别并标记需补查？

### 2. Hedge 词清扫
**全文 grep 这些词，每一个命中都要追问**：
- "应该 / 可能 / 大概 / 估计 / 我记得 / 我觉得 / 似乎 / 大概率 / 八成 / 据我所知"
- "嫌疑 / 怀疑 / 印象中 / 看上去"

每命中一处，writer 必须改成"已验证：X" 或"未验证，建议 Y 后续确认"。

### 3. 弱论证模式识别
**警告模式**：
- "0 hits 所以可删" → **必须**追问：原本干嘛的？历史 commit？删了影响什么？为什么以前要建？
- "上游有 / 上游没有" → 必须追问：上游什么时候开始这样？为什么这么做？我们和上游业务一致吗？
- "默认值就是 X" → 必须追问：为什么默认这样？谁定的？我们的业务和默认假设一样吗？
- "这是惯例" → 必须追问：哪里规定的？文档链接？什么场景下惯例不适用？

### 4. 决策明确性
- 决策类文档（"改 vs 不改"）最后是否给明确建议？
- 是否说了"建议 X 因为 Y"？
- 有没有"两边都对"模糊态度？
- 替代方案有没有列出？每个替代方案的优劣有没有量化对比？

## 来龙去脉层

### 5. 五元素完整性
每个改动 / 决策点必须含五元素：

| # | 元素 | 问题 |
|---|---|---|
| 1 | 当前状态 | 现在是什么样？ |
| 2 | 原本设计意图 | 当初为什么这么设计？ |
| 3 | 历史变化 | 什么时候 / 谁 / 为什么改的？ |
| 4 | 改动后会怎样 | 如果改，会发生什么？ |
| 5 | 不改会怎样 | 如果不改，会发生什么？ |

任何一个元素缺失 → B 级问题。

### 6. 历史溯源
- 关键代码 / 配置的引入 commit 是否查了？
- 引入时的 commit message / 作者 / 上下文是否说清楚？
- 后续是否有修改？修改原因？

### 7. 反向推演
- 如果删 / 改 X，谁先发现问题？什么时候？怎么发现？
- 如果不改 X，未来什么时候会撞墙？撞墙时怎么处理？
- 是否考虑过"撤销"路径？改完发现错了能不能回滚？

## 一致性层

### 8. 文档内一致性
- 关键术语前后用同一个名字吗（不要"分区表" / "GPT 表" / "partition layout file" 混用）？
- 数字前后一致吗（不要这章 4 MB 那章 4096 KB 看起来不同）？
- 决策建议前后一致吗（不要 TL;DR 说 A，正文说 B）？

### 9. 跨文档一致性
- 和 cross-ref 的其他文档（doc N / N+1 / N+2 …）一致吗？
- 如果有冲突，哪个是对的？是否在新文档里明确说"修正之前 doc N 的 X 错误"？

### 10. 与代码 / 实测一致性
- 文档里的 file:line 现在还存在吗（grep 验证）？
- 数字（实测大小 / 占用率）是不是最新的？
- 如果代码已经变了，文档是不是同步更新？

## 评审员视角层

### 11. 多角色 why 测试
对每个目标受众角色，列出**至少 3 个**可能问的"为什么"：

**业务 PM** 视角：
- 为什么现在要做这个改动？
- 改动成本多少？工时？
- 不改会丢多少订单 / 多少客户投诉？

**硬件工程师** 视角：
- 这个改动会不会让硬件量产 yield 下降？
- BootROM / 闭源 blob 真的兼容吗？
- 如果中途断电 / 烧录失败，会不会变砖？

**法务 / 客户支持** 视角：
- 这个改动会不会影响认证（PCI / FCC / CE）？
- 客户合同里有没有约定不能改的部分？
- 这个改动会不会触发客户重新验证 / 重新认证？

**上游 / 平台** 视角：
- 这个改动违反上游推荐的标准设计吗？
- 改完以后还能直接用上游给的工具链吗？
- 上游 SDK 升级时会不会和我们的改动冲突？

文档里能不能答上每个角色的关键问题？

## 风险评估层

### 12. 风险列表完整性
- 列出了所有可见风险（不只是技术风险）？
- 每个风险有"等级（高 / 中 / 低）+ 概率 + 影响 + 缓解措施"四元素？
- 是否考虑了"最坏情况"？

### 13. 灰犀牛 vs 黑天鹅
- **灰犀牛**（明显但被忽视的风险）：是否揭示？
- **黑天鹅**（小概率大影响）：是否预设了应急方案？

### 14. 时间维度
- 当下风险？
- 1 年后风险？
- 5 年后风险（特别对量产硬件）？
- 风险随时间是增大还是减小？

## 视觉表达层（评审员最常忽视的一块 · 必查）

### V1 · 文本墙检测
- 连续 **> 200 字纯文本无图无表** = 必须改
- 决策书 / 复杂技术文档：**每 300 字至少 1 张图或 1 个表**
- 长段落 → 拆 bullet / 表格 / 编号

### V2 · 复杂关系必须有图（不能用文字代替）
以下场景**强制要求视觉化**：

| 信息类型 | 必须用 | 文字代替不行的原因 |
|---|---|---|
| 多模块协作 / 多文件依赖 | **关系图 / 框图** | 文字描述 N 个模块互相依赖，读者要在脑子里建模型 |
| 时序流程（A → B → C → D） | **流程图 / 时序图 / 泳道图** | 多分支决策用文字写就是嵌套 if/else 像迷宫 |
| 改动前后对比 | **before/after 双栏图** 或 **diff 视觉化** | 文字"原本 X，改后 Y"读者要回头比对 |
| 多方案对比（A/B/C） | **决策树 SVG** 或 **方案象限图** | 文字写 N×M 个对比点，注意力涣散 |
| 数据分布 / 占用率 | **柱状图 / 饼图 / 堆叠条** | 数字表格无法快速看出比例和异常值 |
| 时间轴 / 演变史 | **时间轴 SVG** | 文字"X 时引入 Y 时改" 没空间感 |
| 风险评估 | **风险象限（概率 × 影响）** | 表格说"中风险"读者无法快速 triage |
| 决策树（如 IF...THEN...ELSE） | **决策流程图** | 文字写嵌套条件 = 程序员都看得头疼 |
| 文件 / 系统 / 仓库 / 角色架构 | **架构图 / 拓扑图** | 谁连谁谁依赖谁纯文字写最易丢 |
| 量产时间窗 | **甘特图 / 时间轴** | "5-6 周"几个字读者无法 plan 进自己日程 |

### V3 · 决策书的"标配图"清单
任何"改 vs 不改"决策书**至少包含 3 张图**：
1. **当前状态全景图**（让读者一眼看到现状的全貌）
2. **决策树 / 选择路径**（出现 X 条件 → 改；出现 Y 条件 → 不改）
3. **改动影响范围图**（改了 A 会带动 B/C/D，要让读者看见牵连）

可选：
4. 方案对比 quadrant
5. 风险矩阵（impact × likelihood）
6. 时间轴 / 量产甘特

### V4 · 表格也是视觉表达
- 表格 vs 段落：超过 3 行可比较的内容 → 表格
- 表格列 ≤ 7 列（再多就分两个表）
- 表格单元格内容简短，复杂内容拉到 footnote

### V5 · SVG 内嵌的硬性要求
- 内嵌 SVG（不引用外链）—— 文档自包含
- viewBox 设好让 SVG 自适应宽度
- 标签字号 ≥ 11 px（不要 8 px 老花眼）
- 颜色用 anthropic-design 调色板（避免和品牌冲突）
- 关键节点 / 决策菱形 / 流向箭头都要清晰

### V6 · 视觉类型 vs 内容匹配（避免错配）

| 内容 | ✓ 用这个 | ✗ 不要用 |
|---|---|---|
| 父子从属 | 树形图 | 流程图（流向无意义） |
| 时序 / 因果 | 流程图 / 时序图 | 树形图（时间不分明） |
| 互相依赖（多向） | 网状图 / 框图 + 双向箭头 | 流程图（流向单一不自然） |
| 数量对比 | 柱状图 / 表格 | 饼图（除非比例本身是重点） |
| 二维分类（X 轴 + Y 轴） | 象限图 | 表格（无法快速看分布） |

## 形式层

### 15. 禁用词扫描
全文 grep 这些"装 b 黑话"，每个命中都是 C 级问题：

- 矩阵 · 闭环 · 赋能 · 抓手 · 打通 · 拉齐 · 主战场 · 地址簿 · 一张网 · 生态
- 赛道 · 心智 · 体感 · 同频 · 顶层设计 · 破圈 · 护城河 · 降维打击
- 私域 · 公域 · 中台 · 内循环 · 外循环 · 颗粒度 · 方法论 · 对焦 · 卡位
- 复盘 · 沉淀（指经验沉淀，非技术沉淀）

例外（技术含义合法）：字节对齐、代码复用、软件迭代、网络链路。

### 16. 视觉强调
- 关键决策（改 / 不改）有红框 / 大字 / verdict-card 强调吗？
- 关键数据有表格而不是长段落吗？
- 关键对比有 SVG 图 / 对比卡而不是纯文字吗？

### 17. 数字精确性
- 不要"大约 / 几个 / 几天前 / 一段时间" → 改成具体数字 / 具体日期
- 不要"占用率较高" → 改成"占用率 N%"

## 评审员综合印象

最后用 ≤ 100 字总结：
- 文档主要薄弱点是什么？
- writer 下一轮应该重点修改什么？
- 整体距离"可发布"还差几步？

如果是 v1 → 通常会有 5-10 个问题
如果是 v2 → 应该剩 2-3 个问题
如果是 v3 → 应该 0-1 个问题，否则需要回主对话重新对齐需求
