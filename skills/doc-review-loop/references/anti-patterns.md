# 写决策文档的反面教材清单

writer 写稿前必读，reviewer 评审时按这个清单挑刺。每条都对应实战发生过的"被打回重写"案例。

## 反面教材 1 · "0 hits 所以可删"

**症状**：

> "整个代码库 grep `legacy_status` 0 命中，所以这 6 个废弃字段可以从表里删掉。"

**为什么不行**：

只看到"现在没人用"，没看到：
- 当年为什么要加这列？哪个 commit / 什么背景？
- 是从未上线的功能预留，还是历史功能已经废弃？
- 删了之后真的不会有任何隐式依赖（ORM 映射 / 第三方 lib / 下游报表 SQL）？
- 如果未来加新业务想用，能不能通过其他途径实现？

**正确写法**：

> "`legacy_status` 列在 commit `<hash>`（YYYY-MM-DD，作者 X）由内部团队引入，commit message 标注'预留字段'。**'预留'**是关键词 —— 当年 schema 里声明用于承接后续状态机重构，实际业务功能未实现。三方对照：(a) 整个代码库 grep 0 命中（只 schema DDL 一处声明）·(b) 应用层 ORM 模型未映射此列 ·(c) 下游报表 SQL 无引用。删除影响：(1) 生产库历史行如果写过该列的值，迁移跑完这些数据会永久丢失（建议先导一份归档备份确认）；(2) 未来加新业务需要用时，需重新写一版加列迁移（一旦迁移在生产库上跑过 = 回不去 = 得再排一次迁移窗口）。建议：删，理由是当前确认无业务依赖，且清掉这 6 个死列能让表结构和迁移历史更清爽。"

## 反面教材 2 · "应该 / 可能 / 大概"hedge 链

**症状**：

> "应该 max_len == 字段实际长度时写入可能会被拒绝...大概率某 config 调整能解决问题..."

**为什么不行**：

读者无法判断这是"严谨分析后的概率估计"还是"没查直接猜"。结论可信度归零。

**正确写法**：

> "实测：某行的 `name` 值 255 字符 == 列声明 VARCHAR(255)，零余量。数据库是否拒绝 length==limit 边界值取决于版本（公开发行的 vX.x 实测可写，vY.x 部分版本在严格模式下截断报错，未验证 vZ.x+）。如果用户写失败的具体版本是 vN.M，根据官方文档描述确实是严格 less-than，需要测试。"

把不确定的部分**显式标记**为"需验证 X"，不混在结论里。

## 反面教材 3 · 前后矛盾

**症状**：

文档第 2 章："本项目沿用旧 schema，所以不能删 `legacy_status` 列（必须保留）"
文档第 6 章："建议删掉 `legacy_status` 列"

**为什么不行**：

读者一眼看出矛盾，对整篇文档的可信度打对折。

**正确写法**：

如果前后有不同建议，必须在两处都加上"前置条件"：
- 第 2 章："如果该迁移已在生产库上跑过 N 次，则不能再改这张表结构（包括删列）。下面以这个为前提。"
- 第 6 章：（不会出现，因为前置条件已锁死）

或者：
- 第 2 章："迁移上线前可删 / 上线后不可删"
- 第 6 章："基于'迁移未上线'前提，建议..."

## 反面教材 4 · "可改可不改"

**症状**：

> "改 (configA=N1, configB=N2) 有好处但也有代价 ... 不改也行 ... 看你怎么选。"

**为什么不行**：

决策书的目标就是给决策。"看你怎么选"等于没决策。

**正确写法**：

> "**建议改**。理由：(1) 当前数据 X / 上限 Y 离上限 Z 单位，下次上量几乎必撞容量上限报错；(2) 改 configA=N 给 X 增长 M 余量，单次 5 分钟成本；(3) 该方案在另一个同构服务上已线上验证。**或者不改的明确触发条件**：如果运维团队反馈某能力实战证明有用（拿出具体案例），保持原配置。"

明确建议 + 触发条件下的反向决策。

## 反面教材 5 · 缺来龙去脉

**症状**：

> "建议把 `name` 列从 VARCHAR(255) 扩到 VARCHAR(512)。"

**为什么不行**：

评审员问"为什么 255？为什么 512？为什么不是 384？" → 答不上。

**正确写法**：

包含五元素：
- **当前状态**：列 VARCHAR(255) · 实测最长的一行 name 恰好 255 字符，撞顶
- **原本设计意图**：初版需求里 name 只放短标识，255 余量充足。这是老 schema 的默认。
- **历史变化**：需求迭代后 name 改承接完整展示名，实测已有行到 250 字符。
- **改动后会怎样**：扩 512 后，name 可放到 512 字符，列有充足富余，写入边界值消失。
- **不改会怎样**：下次业务再加几个字符即超上限 → 写入截断 / 报错；OR 严格模式下直接 reject。
- **为什么 512 不是 384**：同类字段的公司规范推荐用 2 的幂（256/512/1024）方便统一，512 覆盖当前需求且留足余量。384 偏少且不符合规范。

## 反面教材 6 · 装 b 黑话

**症状**：

> "通过打通 X 业务和上游生态的链路矩阵，赋能下游 schema 改造的闭环..."

**为什么不行**：

工程师看了就烦。可读性归零。

**正确写法**：

> "把本服务的 schema 和共享库的数据模型统一，覆盖在线读写 / 离线报表 / 数据同步三个场景。"

## 反面教材 7 · 数字模糊

**症状**：

> "占用率较高，离上限不远了，建议尽早处理。"

**为什么不行**：

"较高" / "不远" / "尽早" 都是相对词，无法决策。

**正确写法**：

> "`audit_log` 表实测 2,089,472 行 / 上限 2,097,152 行 = 99.6%。剩余 7,680 行增长空间。按近期日均新增 ~1,200 行估算，约 6 天内即触发容量上限报错。"

## 反面教材 8 · 漏闭源依赖风险

**症状**：

> "改 schema 没问题，应用代码都是我们自己的，跑一遍单测就能验证。"

**为什么不行**：

忽略第三方闭源 SDK / 外部下游系统对表结构的隐式假设。

**正确写法**：

> "改 schema 影响面：(1) 应用代码：自有仓库，跑单测验证；(2) ORM 映射：自有工具，改模型即可；(3) **第三方闭源 SDK**：某数据同步 SDK 对列名 / 类型的硬编码假设**未公开文档**。证据：该 SDK 所有集成样例里都直接读 `legacy_status` 列（疑似硬编码字段名）。建议：改之前先开技术支持工单确认或在预发环境完整跑一遍端到端同步测试。"

## 反面教材 9 · 不评估反向 / 撤销路径

**症状**：

> "改完 schema 跑回归测试，通过就 ship。"

**为什么不行**：

如果迁移跑完发现第三方 SDK 不兼容，能撤销吗？怎么撤销？要不要先准备回滚迁移脚本？

**正确写法**：

> "改完流程：(1) 预发环境全量回归；(2) 如发现兼容问题，撤销路径 = 迁移未上线时直接 git revert 改回 = 0 损失；(3) 如迁移已在预发库上跑过，撤销 = 写一版回滚迁移把列加回来 + 从归档补数据（数据可能已丢失，要评估）；(4) 如迁移已在生产库上跑过且删了列，删掉的数据不可逆，只能修复前向；(5) 因此分阶段 gate：预发验通过 → 灰度一个副本库验通过 → 再上主库。"

## 反面教材 10 · 单维度看问题

**症状**：

> "代码 grep 0 hits，可以删这一列。"

**为什么不行**：

只看了"代码维度"，没看：
- 业务维度：业务团队会不会有规划用？
- 合规维度：客户合同 / 数据留存要求里有没有约定必须保留？
- 历史维度：哪个版本引入？删了影响哪些已上线的下游报表？
- 工具维度：第三方闭源 SDK 有没有依赖？

**正确写法**：

按 4 维度（编译 / 运行时 / 业务 / 合规）逐项评估，每个维度给出结论 + 证据。

## 反面教材 11 · 字墙没图

**症状**：

> （3000 字纯文本 + 5 个表格 + 0 张图）
> "改 X 会影响 A、B、C、D；A 又依赖 E、F；E 通过 G 触发 H；时序上 T0 写 X，T1 迁移 Y，T2 验证 Z..."

**为什么不行**：

读者头脑里在重建你描述的关系图，每多 1 个节点就多 1 个负担。10 个节点写完读者已经懵了。视觉化能 5 秒告诉读者的结构，文字要 5 分钟。

**正确写法**：

每个"复杂关系"或"多步流程"或"多方案对比"或"演变史"，用 inline SVG 画出来。决策书至少 3 张图：当前状态全景 / 决策树 / 改动影响范围。

**抽象案例**：

某后端服务 schema 重构项目的决策书，43 KB HTML，1243 行，包含 verdict-card / 27 行的表结构清单 / 15 张表格 / 4 个 callout，但 **0 张 SVG 流程图 / 决策树 / 关系图**。reviewer 通过了，但 PM 审阅时一眼指出"密密麻麻的字，字表达不是不可以，但你不少都可以用流程图和框图去表达清晰"。

教训：**reviewer checklist 漏了"视觉表达"维度**，已升级到 V1-V6 6 项硬检查。

## 反面教材 12 · 没和关联文档交叉对照

**症状**：

写新文档没核对已有相关文档，结果新文档和老文档结论冲突，或者重复证据但版本不一致。

**为什么不行**：

读者想顺着读 N 份相关文档时，发现互相打脸或者数字微差对不上，对整套文档信任崩塌。

**正确写法**：

写新文档前**必查**：
1. 列出该项目已有相关文档（grep 同目录的 .html / .md）
2. 读关键章节看结论 / 数据 / 术语
3. 新文档若发现老文档错的：明确写"修正 doc N 的 X 错误"
4. 老文档若需要更新：在新文档里 cross-ref 标"建议同步更新 doc N"
5. 关键术语 / 数字 / 决策建议跨文档保持一致

**抽象案例**：

4 份关联文档，初稿阶段没核对一致性。早期 doc 写"项目沿用旧 schema（不能改 `orders` 表结构）"是默认假设；后来 doc 实测发现某些列是死列可以删、反推该表可以安全重构。两份文档默认假设不一致，用户阅读时发现矛盾。教训：**新文档发布前必须跑一遍"关联文档一致性审计"**。

## reviewer 的兜底问题

如果不知道挑什么刺，问这 3 个万能问题：

1. **"为什么不是反过来？"** —— 如果建议改，问"为什么不是不改？"；如果建议保持，问"为什么不是改？"
2. **"我是 [其他角色] 我会问什么？"** —— 法务 / 业务 / 上游对接 / 客户的视角能问出什么？
3. **"5 年后回头看，这个决策会不会被骂？"** —— 想象未来翻这份文档的人，会觉得当时考虑周全吗？
