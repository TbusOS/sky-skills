# 写决策文档的反面教材清单

writer 写稿前必读，reviewer 评审时按这个清单挑刺。每条都对应实战发生过的"被打回重写"案例。

## 反面教材 1 · "0 hits 所以可删"

**症状**：

> "整个代码库 grep `partitionA` 0 命中，所以这 6 个分区可以删除。"

**为什么不行**：

只看到"现在没人用"，没看到：
- 当年为什么要建？哪个 commit / 什么背景？
- 是从未实现的功能预留，还是历史功能已经废弃？
- 删了之后真的不会有任何隐式依赖（jar 包 / 闭源 blob / 第三方 lib）？
- 如果未来加新业务想用，能不能通过其他途径实现？

**正确写法**：

> "partitionA 在 commit `<hash>`（YYYY-MM-DD，作者 X）由内部团队引入，commit message 标注'预留分区'。**'预留'**是关键词 —— 当年配置文件里声明用于预防 OTA wipe，实际业务功能未实现。三方对照：(a) 整个代码库 grep 0 命中（只配置文件一处声明）·(b) 出厂烧录路径不烧此分区 ·(c) 上游 SDK 变体都无此分区。删除影响：(1) 历史设备如果出厂时往这分区写过数据，新固件会丢失（建议查历史烧录记录确认）；(2) 未来加新业务需要用时，需重新申请扩分区表（出货后做不了 = 必须用别的存储）。建议：删，理由是当前确认无业务依赖且 N MB 死空间换其他撞顶分区扩展空间更紧迫。"

## 反面教材 2 · "应该 / 可能 / 大概"hedge 链

**症状**：

> "应该 4 MB == 4 MB 时烧录工具可能会拒绝...大概率某 config 调整能解决问题..."

**为什么不行**：

读者无法判断这是"严谨分析后的概率估计"还是"没查直接猜"。结论可信度归零。

**正确写法**：

> "实测：image 4194304 字节 == 分区 4194304 字节，零余量。烧录工具是否拒绝 size==partition 边界值取决于工具版本（公开发行的 vX.x 实测可烧，vY.x 部分版本拒绝，未验证 vZ.x+）。如果用户烧失败的具体版本是 vN.M，根据官方文档描述确实严格 less-than，需要测试。"

把不确定的部分**显式标记**为"需验证 X"，不混在结论里。

## 反面教材 3 · 前后矛盾

**症状**：

文档第 2 章："本项目沿用旧分区表，所以不能扩某分区到 8 MB（必须保持 4 MB）"
文档第 6 章："建议把某分区扩到 8 MB"

**为什么不行**：

读者一眼看出矛盾，对整篇文档的可信度打对折。

**正确写法**：

如果前后有不同建议，必须在两处都加上"前置条件"：
- 第 2 章："如果产品已量产出货 N 万台，则不能改分区表（包括分区大小）。下面以这个为前提。"
- 第 6 章：（不会出现，因为前置条件已锁死）

或者：
- 第 2 章："出货前可改 / 出货后不可改"
- 第 6 章："基于'未出货'前提，建议..."

## 反面教材 4 · "可改可不改"

**症状**：

> "改 (configA=N1, configB=N2) 有好处但也有代价 ... 不改也行 ... 看你怎么选。"

**为什么不行**：

决策书的目标就是给决策。"看你怎么选"等于没决策。

**正确写法**：

> "**建议改**。理由：(1) 当前数据 X / 上限 Y 离上限 Z 单位，下次升级几乎必撞 build fail；(2) 改 configA=N 给 X 增长 M 余量，单次 5 分钟成本；(3) 该方案在上游某变体已生产验证。**或者不改的明确触发条件**：如果硬件团队反馈某能力实战证明有用（拿出具体案例），保持原配置。"

明确建议 + 触发条件下的反向决策。

## 反面教材 5 · 缺来龙去脉

**症状**：

> "建议把分区从 4 MB 扩到 8 MB。"

**为什么不行**：

评审员问"为什么 4 MB？为什么 8 MB？为什么不是 6 MB？" → 答不上。

**正确写法**：

包含五元素：
- **当前状态**：分区 4 MB · 实测 image 4 MB exact 撞顶
- **原本设计意图**：旧 SDK 时代相应模块体积约 1 MB，4 MB 余量充足。这是上游老 SDK 的默认。
- **历史变化**：新 SDK 升级后模块加多项功能，实测已 2040 KB。
- **改动后会怎样**：扩 8 MB 后，模块可长到 4096 KB，分区有 4 MB 富余，烧录边界值消失。
- **不改会怎样**：下次 SDK 升级模块增 7.5 KB 即超上限 → build fail；OR 烧录工具严格版本 reject。
- **为什么 8 MB 不是 6 MB**：上游某变体已采用 8 MB（参考设计），且 8 MB 是 4 MB 整数倍便于分区对齐。6 MB 偏少且非 4 MB 对齐。

## 反面教材 6 · 装 b 黑话

**症状**：

> "通过打通 X 业务和上游生态的链路矩阵，赋能下游分区改造的闭环..."

**为什么不行**：

工程师看了就烦。可读性归零。

**正确写法**：

> "把项目分区方案和上游 SDK 设计统一，覆盖 OTA 升级 / 出厂烧录 / fastboot 三个场景。"

## 反面教材 7 · 数字模糊

**症状**：

> "占用率较高，离上限不远了，建议尽早处理。"

**为什么不行**：

"较高" / "不远" / "尽早" 都是相对词，无法决策。

**正确写法**：

> "image 实测 2,089,472 字节 / 上限 2,097,152 字节 = 99.6%。剩余 7,680 字节增长空间。按近期升级单次增 ~50 KB 估算，下次 SDK 升级（预计 N 个月内）即触发 build fail。"

## 反面教材 8 · 漏闭源工具风险

**症状**：

> "改分区表没问题，build 系统都是开源的，跑 make 就能验证。"

**为什么不行**：

忽略上游闭源 blob（loader binary / DDR init / 烧录工具）对分区布局的隐式假设。

**正确写法**：

> "改分区表影响面：(1) build 系统：开源，跑 make 验证；(2) AVB 签名：开源工具，重签即可；(3) **闭源 blob**：上游预编译 loader 对分区起点 / 大小的硬编码假设**未公开文档**。证据：上游所有变体的某分区起点都是 0x4000（疑似硬编码常量）。建议：改之前先开技术支持工单确认或在 verify 设备完整烧测。"

## 反面教材 9 · 不评估反向 / 撤销路径

**症状**：

> "改完分区表后跑回归测试，通过就 ship。"

**为什么不行**：

如果改完发现闭源 blob 不兼容，能撤销吗？怎么撤销？要不要先准备 fallback 镜像？

**正确写法**：

> "改完流程：(1) verify 板子全量烧测；(2) 如发现兼容问题，撤销路径 = git revert 改回 + 重 build + 重烧 verify 设备验证 = 0 损失；(3) 如已经推到生产线 N 台 PCBA 测试，撤销 = N 台板子重烧（每台 30 分钟，N=20 时 10 工时成本）；(4) 如已批量出货，撤销不可行，只能修复前向；(5) 因此分阶段 gate：verify 验通过 → PCBA 试产 50 台 → 验通过再扩产。"

## 反面教材 10 · 单维度看问题

**症状**：

> "代码 grep 0 hits，可以删某分区。"

**为什么不行**：

只看了"代码维度"，没看：
- 业务维度：业务团队会不会有规划用？
- 合规维度：客户合同 / 认证文档里有没有约定必须存在？
- 历史维度：哪个版本引入？删了影响哪些已发出的固件？
- 工具维度：闭源工具有没有依赖？

**正确写法**：

按 4 维度（编译 / 运行时 / 业务 / 合规）逐项评估，每个维度给出结论 + 证据。

## 反面教材 11 · 字墙没图

**症状**：

> （3000 字纯文本 + 5 个表格 + 0 张图）
> "改 X 会影响 A、B、C、D；A 又依赖 E、F；E 通过 G 触发 H；时序上 T0 写 X，T1 烧 Y，T2 验证 Z..."

**为什么不行**：

读者头脑里在重建你描述的关系图，每多 1 个节点就多 1 个负担。10 个节点写完读者已经懵了。视觉化能 5 秒告诉读者的结构，文字要 5 分钟。

**正确写法**：

每个"复杂关系"或"多步流程"或"多方案对比"或"演变史"，用 inline SVG 画出来。决策书至少 3 张图：当前状态全景 / 决策树 / 改动影响范围。

**抽象案例**：

某商用嵌入式终端 SDK 升级项目的分区决策书，43 KB HTML，1243 行，包含 verdict-card / 27 分区表 / 15 张表格 / 4 个 callout，但 **0 张 SVG 流程图 / 决策树 / 关系图**。reviewer 通过了，但 PM 审阅时一眼指出"密密麻麻的字，字表达不是不可以，但你不少都可以用流程图和框图去表达清晰"。

教训：**reviewer checklist 漏了"视觉表达"维度**，已升级到 V1-V6 6 项硬检查。

## 反面教材 12 · 没和关联文档交叉对照

**症状**：

写新文档没核对已有相关文档，结果新文档和老文档结论冲突，或者重复证据但版本不一致。

**为什么不行**：

读者想顺着读 N 份相关文档时，发现互相打脸或者数字微差对不上，对整套文档信任崩塌。

**正确写法**：

写新文档前**必查**：
1. 列出该项目已有相关文档（grep 同目录的 .html / .md）
2. 读关键章节看结论 / 数据 / 术语
3. 新文档若发现老文档错的：明确写"修正 doc N 的 X 错误"
4. 老文档若需要更新：在新文档里 cross-ref 标"建议同步更新 doc N"
5. 关键术语 / 数字 / 决策建议跨文档保持一致

**抽象案例**：

4 份关联文档，初稿阶段没核对一致性。早期 doc 写"项目沿用旧分区表（不能扩某分区到 8 MB）"是默认假设；后来 doc 实测发现某些分区是死分区可以删反推目标分区可以扩。两份文档默认假设不一致，用户阅读时发现矛盾。教训：**新文档发布前必须跑一遍"关联文档一致性审计"**。

## reviewer 的兜底问题

如果不知道挑什么刺，问这 3 个万能问题：

1. **"为什么不是反过来？"** —— 如果建议改，问"为什么不是不改？"；如果建议保持，问"为什么不是改？"
2. **"我是 [其他角色] 我会问什么？"** —— 法务 / 业务 / 上游对接 / 客户的视角能问出什么？
3. **"5 年后回头看，这个决策会不会被骂？"** —— 想象未来翻这份文档的人，会觉得当时考虑周全吗？
