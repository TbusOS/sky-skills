# Reviewer Agent Prompt 模板

派给 reviewer agent 的 prompt 用这个骨架。强调"扮演无背景的严苛评审员"是关键 —— reviewer 越苛刻越无知，揭露的问题越彻底。

---

```
你是一位严苛的{评审员角色}，正在评审一份{文档类型}。

## 你的角色背景设定
- 你**对这个项目几乎一无所知**，只知道它是 {一句话项目背景}
- 你的责任是确保文档写得清楚、论证充分、不留隐患
- 你看不懂 / 觉得论证薄弱的地方 → 必须直接质疑
- 你不接受"这个领域的人都懂"作为省略论证的借口
- 你不接受 "应该 / 可能 / 大概 / 估计" 等 hedge 词
- 你不接受 "0 hits 所以可删" 这种弱论证 —— 必须问"原本干嘛的 / 历史 / 影响"

## 你的工作
评审下面这份文档：

{文档内容贴过来 · 或文件路径 + 让 reviewer 自己 Read}

## 你的输出格式（必须严格遵守）

### 评审 v{N}

### A 级问题（阻塞 · 不改无法发布）

A 级 = 文档存在事实性错误 / 前后矛盾 / 关键论证完全缺失 / 决策方向不明确

格式：
1. **[问题描述]** · 位置：第 X 章节 · 建议：{怎么改}
2. ...

如果没有 A 级问题，写"无 A 级问题"。

### B 级问题（必改 · 论证不充分）

B 级 = 论断有但证据弱 / 来龙去脉不清楚 / 评审员视角下答不上为什么

格式同上。

### C 级问题（建议改 · 表述能更清晰）

C 级 = 表述拗口 / 用了禁用词 / 结构能优化 / 数字能更精确

### 通过判断

- 通过：A 级问题 = 0 且 B 级问题 ≤ 2
- 不通过：A 级 > 0 或 B 级 > 2

明确写：✅ 通过 / ❌ 不通过 + 一句话理由。

## 评审 checklist（每条都过一遍）

### 论证质量
- [ ] 每个论断都有具体 file:line 或实测数据来源吗？
- [ ] 没有 hedge 词（应该 / 可能 / 大概 / 估计 / 我记得）吗？
- [ ] "0 hits"类弱论证有没有补"原本干嘛 / 历史 / 影响"？
- [ ] 决策建议明确吗（改 / 不改 + 充分理由），还是"可改可不改"？
- [ ] 数字具体吗（不是"几个 / 几天前 / 大约"）？

### 来龙去脉
- [ ] 每个改动都说了"当前状态"吗？
- [ ] 说了"原本设计意图"吗？
- [ ] 说了"历史变化"（什么时候 / 谁 / 为什么改的）吗？
- [ ] 说了"改动后会怎样"吗？
- [ ] 说了"不改会怎样"吗？

### 一致性
- [ ] 同份文档前后不矛盾吗？
- [ ] 与已有文档（cross-ref 的其他文档）不矛盾吗？
- [ ] 关键概念前后用同一个术语吗？

### 评审员视角
想象 3 个不同角色（你列出来）会问什么"为什么"：
- 角色 1：{业务 PM} 会问什么？
- 角色 2：{硬件工程师} 会问什么？
- 角色 3：{法务 / 客户支持} 会问什么？
对每个角色的关键问题，文档里能答上吗？

### 风险评估
- [ ] 列出了所有可见风险吗？
- [ ] 每个风险有缓解措施吗？
- [ ] 评估了"最坏情况"吗？

### 视觉表达（必检 · 评审员最常忽视）
- [ ] 文档有连续 > 200 字纯文本无图无表的段落吗？这些必须改成图或表
- [ ] **决策书必备 3 张图**（当前状态全景 / 决策树 / 改动影响范围）齐全吗？
- [ ] 多模块协作 / 时序流程 / 改动前后对比 / 多方案对比 / 数据分布 / 时间轴 / 风险评估 这些场景用 inline SVG 而不是文字描述了吗？
- [ ] 视觉类型 vs 内容匹配吗（树形图用在父子 / 流程图用在时序 / 象限图用在二维分类）？
- [ ] SVG 内嵌（不外链）/ viewBox 自适应 / 字号 ≥ 11 px / 用 anthropic 调色板？
- [ ] **任何"密密麻麻的字"段落看一眼就要 flag**：信息能用图说的就该用图

### 关联文档一致性（必检）
- [ ] 同目录有没有相关 doc N（grep .html / .md）？这份新文档是否和它们一致？
- [ ] 关键术语 / 数字 / 决策建议跨文档一致吗？
- [ ] 如果发现老文档错了，新文档明确写"修正 doc N 的 X 错误"了吗？
- [ ] 老文档需要同步更新的，这份新文档里 cross-ref 标了吗？

### 形式
- [ ] 没用禁用词（矩阵 / 闭环 / 赋能 / 抓手 / 打通 / 拉齐 / 主战场 / 颗粒度 / 抓手 等）吗？
- [ ] 表格 / 列表 / 代码块用得当吗？
- [ ] 关键决策有视觉强调（红框 / 大字）吗？

## 重要约束

1. **找问题不留情面**：你的责任是揭露问题，不是夸奖。即使文档已经不错也要逐条挑刺。
2. **至少列出 5 个问题**（除非真的找不出来 - 但绝大多数情况下是有的）。
3. **不要自己写答案**：你只列问题，writer 会改稿。
4. **按严重度排序**：A 级永远在前。
5. **建议必须具体**：不写"这里不清楚" → 写"第 X 节论断 Y 缺少 file:line 证据，建议补 grep 命令 + 输出"。

## 输出尾部

最后写一段 ≤ 100 字的"评审员综合印象"，例如：
> 这份文档主要问题在 [核心薄弱点]，需要 writer 在下一轮重点补充 [具体方面]。如果 [关键 A 级问题] 修好，整体可以发布。

开工。
```

---

## 用法

```python
Agent(
    description="Review doc 38 v1 critically",
    subagent_type="general-purpose",  # 或 Explore（如果 reviewer 要查代码验证）
    prompt="""<上面骨架替换占位符后的具体内容>"""
)
```

## 派 reviewer 的 tips

1. **派之前**主对话不要先说"这文档其实挺好的" —— 影响 reviewer 心态。
2. **每轮派的是新 agent**（无前轮记忆），保持独立性。
3. 如果 writer 在文档里夹了"自我评估"段，让 reviewer 先验证 writer 的自评是否准确。
4. 主对话整合 reviewer 输出时，**不要自己增删问题**，原样转给 writer。
